\subsection{Community detection algorithms}

We use three different community detection algorithms to characterize the community structure of the networks: Infomap, the Louvain method and the Agglomerative Greedy algorithm. In this section we briefly outline the theoretical foundations of each approach.

\subsubsection{Infomap}

The Infomap algorithm \cite{rosvall2009map} is an information-theoretic method for community detection. Unlike modularity maximization approaches which use the structural connection of the network, Infomap identifies communities based on network dynamics and information flow.

To model this flow, the algorithm uses a random walker navigating through the network's links. The main idea is that if a network has a well defined community structure, the random walker will spend extended periods of time trapped within certain clusters of nodes before going to other regions of the network. 

By finding the most compact way to describe the random walker's path, Infomap uses data compression to show the underlying flow of information in the network. To do so, the algorithm uses a two-level coding scheme:

\begin{enumerate}
    \item \textbf{Module Codes:} Nodes are partitioned into modules (communities). A unique index code is assigned to each module to track when the random walker transitions from one community to another.
    \item \textbf{Node Codes:} Within each module, nodes are assigned local codes. Because these local codes can be reused across different modules, the overall description of the random walker's path requires significantly less information.
\end{enumerate}

The core of the algorithm is the Map Equation, which calculates the theoretical minimum description length (in bits) required to track the path of the random walker for any given network partition. Infomap evaluates different node partitions and searches for the one that minimizes the Map Equation. The partition that gets the shortest description length represents the optimal community structure, as it best captures the regions where flow persists the longest.

For the implementation of this algorithm, we use the \href{https://mapequation.github.io/infomap/python/}{Infomap Python API}. It provides an implementation of the map equation framework.


\subsubsection{Louvain}

The Louvain method \cite{blondel2008fast} is a popular heuristic algorithm designed to extract the community structure of large networks by maximizing a metric known as Modularity ($Q$). Modularity measures the density of links inside communities compared to the expected density of links between them in a random network. A higher modularity score indicates a more defined community structure.

Unlike flow-based algorithms like Infomap, Louvain focuses entirely on the structural topology of the network. The algorithm operates in a greedy, iterative manner in two recurring phases:

\begin{enumerate}
    \item \textbf{Local optimization:} Initially, every node is assigned to its own community. The algorithm then evaluates each node and calculates the change in modularity if that node was moved into a neighboring community. The node is then placed into the community that gives the highest increase in modularity. This process is performed sequentially for all nodes, until no more node movements can improve the modularity.
    \item \textbf{Network aggregation:} Once the local optimization is done, the algorithm builds a new macroscopic network. In this new network, the communities found in the first phase become the new \say{nodes} and the links between them are formed by adding the weights of the edges between the original nodes in those respective communities.
\end{enumerate}

These two phases are repeated iteratively until the network cannot be condensed anymore and the modularity reaches its maximum value. This process makes the algorithm very computationally efficient and also provides a hierarchical community structure.

We use the \texttt{louvain\_communities} function provided by the \href{https://networkx.org/documentation/stable/reference/algorithms/generated/networkx.algorithms.community.louvain.louvain_communities.html}{NetworkX library in Python}.


\subsubsection{Agglomerative Greedy algorithm}

The Agglomerative Greedy algorithm \cite{newman2004fast, clauset2004finding} is another approach based on modularity maximization. However, unlike the Louvain method which relies on moving individual nodes to optimize modularity locally, this algorithm uses a bottom-up hierarchical clustering strategy. 

The algorithm begins with an initial state where every node in the network is assigned to its own community. From this point, it goes through an iterative agglomerative process. At each step, the algorithm evaluates all pairs of connected communities and merges the pair that results in the largest possible increase (or smallest decrease) in the overall modularity ($Q$) of the network. This merging process is repeated continuously, building larger and larger communities until all nodes in the network belong to a single community. The optimal community structure is then determined by selecting the specific partition from the merging history that corresponds to the maximum modularity value.

For the implementation of this algorithm, we use the \texttt{greedy\_modularity\_communities} function from the \href{https://networkx.org/documentation/stable/reference/algorithms/generated/networkx.algorithms.community.modularity_max.greedy_modularity_communities.html}{Python NetworkX library}.


\subsection{Analysis of the community detection results}

From $p_{rr} \approx 0.15$ onwards, the three algorithms converge in both metrics. They identify the ground-truth value of 5 communities and follow an identical increasing modularity curve, reaching $Q \approx 0.7$ at $p_{rr} = 1.0$. This confirms that once the internal block structure becomes sufficiently dense, all three methods are able to correctly identify the community structure of the network.

Modularity alone is an not a reliable indicator of true community structure due to the phenomenon of \say{spurious communities}. Our results clearly demonstrate this: when $p_{rr}$ approaches 0.0, the network behaves as an Erdös-Rényi random graph with no actual communities. Yet, modularity maximization algorithms like Louvain and Greedy still report relatively high scores (approximately $Q = 0.45$). They achieve this by exploiting normal statistical fluctuations in the random distribution of edges, artificially grouping them to maximize the modularity function. Therefore, a score like $Q = 0.4$ does not guarantee a meaningful community structure, as a purely random network can produce that exact same value entirely by chance.

For very low values of the intra-block connection probability ($p_{rr} \approx 0.0$), the network does not show a defined topological structure and behaves as a random graph. This lack of communities is reflected in the evaluation metrics (Figures~\ref{fig:jaccard}, \ref{fig:norm_mutual_info}, and \ref{fig:norm_var_information}): both the Jaccard Index and Normalized Mutual Information (NMI) are close to 0.0, while the Normalized Variation of Information (NVI) is near 1.0. 

Visually, the network at $p_{rr} = 0.02$ (Figures~\ref{fig:prr_002_infomap}, \ref{fig:prr_002_louvain}, and \ref{fig:prr_002_greedy}) confirm this. Nodes in each of the five clusters are assigned to different communities, which is shown by a mixture of colors in each of the groups. At this point, the algorithms are grouping nodes into spurious communities rather than identifying true structural patterns.

As the internal connection density increases ($0.05 < p_{rr} < 0.3$), the blocks begin be slighlty more defined, but the algorithms show different levels of ability to capture the real underlying structure. 

On the one hand, Infomap demonstrates the best transition. After a brief interval around $p_{rr} = 0.08$ where it collapses all nodes into a single giant community (causing a drop in NMI to 0.0, as seen in Figure~\ref{fig:norm_mutual_info}), it then achieves perfect partition accuracy, reaching a Jaccard Index and NMI of 1.0 at around $p_{rr} = 0.12$. Louvain follows closely, reaching perfect accuracy near $p_{rr} = 0.15$. In contrast, the Agglomerative Greedy algorithm is the worst one in this aspect, requiring a density of almost $p_{rr} = 0.3$ to perfectly identify the ground truth communities.

These differences between the algorithms are supported by the network snapshots at $p_{rr} = 0.16$. At this threshold, both Infomap (Figure~\ref{fig:prr_016_infomap}) and Louvain (Figure~\ref{fig:prr_016_louvain}) display uniformly colored, which correlate with their perfect metric scores. However, the Greedy algorithm's visualization (Figure~\ref{fig:prr_016_greedy}) still shows some misclassified nodes.

Finally, for $p_{rr} \ge 0.3$, all three algorithms converge to perfect scores. The Jaccard Index and NMI flatten at their optimal values of 1.0, and the NVI remains at its minimum of 0.0, all of them until $p_{rr} = 1.0$. The visual representation at $p_{rr} = 1.0$ (Figures~\ref{fig:prr_100_infomap}, \ref{fig:prr_100_louvain}, and \ref{fig:prr_100_greedy}) fully corroborates this, displaying five perfectly partitioned communities with zero misclassifications, for all three methods. This confirms that once the internal links become dense enough, the community structure is so clear that even the algorithm which takes longest to converge, can identify the ground truth communities without any errors.

\subsection{Detected communities and their composition}
 
Both the unweighted and the weighted versions of the network generate 6 communities under the Louvain algorithm. Despite this identical match in the number of communities, the actual composition of those communities differs between the two cases (especially in the higher grades), and understanding the nature of those differences requires reflecting on what the algorithm can and cannot be expected to recover.
 
A key observation is that the algorithm detects the communities at the grade level rather than at the class level. In all detected communities, the two groups of the same school year (e.g.\ 2A and 2B, or 4A and 4B) are merged into a single community. This is does not necessarily mean that the algorithm fails, but it can be interpreted as a consequence of the social structure of the school itself. It seems that students from the two different groups within the same grade share break times and more common spaces, and thus accumulate a higher number of interactions. The algorithm, optimizing for modularity, correctly identifies that the boundary between groups from the same grade is weaker than the boundary between different grade levels. The notable exception is the first grade (1A and 1B), which appear as two separate communities in both the weighted and unweighted cases. This is consistent with the visual inspection of the original network (Figure~\ref{fig:primary_school_communities}, left), where the red and green groups are more spatially separated, suggesting that 1st-graders interact much more exclusively within their own class than older children do. Indeed, based on our experience at school, even without having performed a scientific study, students tend to interact more with other classes as they get older. This is also reflected in the network, where the separation between grades and classes becomes blurrier in higher courses, except for the strong distinction between 1st, 2nd, 3rd and 4th, 5th grades, which is likely explained by their different lunch times~\cite{stehle2011high}.
 
A further structural observation concerns the teachers. The metadata assigns each class teacher to the generic ``Teachers'' group, but the detected communities consistently distribute teachers across the student communities rather than grouping them together. This comes as no big surprise, as each teacher spends the majority of their time with the students of their own class. Their strongest ties are therefore within their assigned class community, so the algorithm correctly assigns each teacher to their respective class. Because the Louvain method assigns nodes based on the pattern of their connections rather than on any predefined label, it produces a more sociologically meaningful result than the original metadata grouping in this case.
 
Figures~\ref{fig:community_composition_unweighted} and~\ref{fig:community_composition_weighted} show the full composition of the detected communities in terms of the original school groups.

In the unweighted case (Figure~\ref{fig:community_composition_unweighted}), communities 0 and 1 recover the first-grade classes 1A and 1B as separate groups ($\sim$23--26 nodes each). This corresponds to the only grade-level split observed. Communities 3 and 4 are each dominated by a single grade: class 2B and class 3A, respectively (as indicated by their colors), with each community also containing the complementary group of that grade (2A into community 3, and 3B into community 4) along with the corresponding teachers. Community 7 is composed almost entirely of class 4B ($\sim$24 nodes). In sharp contrast, community 9 is very large ($\sim$70 nodes) and highly heterogeneous, aggregating classes 4A, 5A, 5B and the Teachers together. The fact that 4th and 5th grade (and the remaining teachers) are all into a single large community points to a specific weakness of the unweighted approach. In the original network, the spatial boundary between grades 4 and 5 is already less visually clear than those separating lower grades, and without contact durations separating these groups apart is not possible.

The main difference lies in the upper grades. The large heterogeneous community 9 from the unweighted analysis is now split into community 6 ($\sim$44 nodes, dominated by classes 4A and 4B) and community 9 ($\sim$50 nodes, dominated by classes 5A, 5B and Teachers). By incorporating contact durations, the algorithm has more information to identify that the 4th-grade students (despite having many connections to 5th-graders) interact more intensively within their own grade, making the inter-grade boundary distinguishable.

\subsection{Why do weights matter?}
 
The comparison above reveals two complementary aspects of what edge weights contribute to community detection in social interaction networks.
 
The first is a matter of resolution. In the unweighted graph, every recorded contact, despite it being brief, counts equally. A spontaneous encounter in a corridor between a student from 4th grade and another from 5th grade contributes as much to the community structure as hours of in-class interaction. As a result, the accumulated cross-grade links between the upper school years are comparable to within-grade links, and the modularity gain of splitting them is negligible. The weighted version corrects for this, since within-grade contacts accumulate far more total duration than cross-grade ones. Therefore, the algorithm can detect the grade boundary where the unweighted version is not able to.
 
The second aspect concerns granularity. The fact that the algorithm merges the two sections of each grade into a single community is a property of the data, not an artifact of ignoring weights. This is directly supported by the contact matrices reported by Stehlé et al.\ \cite{stehle2011high}, who observe that \textit{``each class has more contacts with the other class of the same grade than with any other class''}, and that the contact matrices reveal \textit{``two-by-two light blocks around the diagonal''} reflecting high contact numbers and durations between classes from the same grade. Weights do not change this conclusion because the ratio of within-group to between-group contact time is not large enough to cross the modularity threshold needed for a further split. We therefore believe that the algorithm operating correctly as it finds the partition that best reflects the actual intensity of social interactions in the data.

\section{A note on the alignment of communities}

When comparing community-detection results across different networks (e.g.\ for different values of $p_{rr}$) or across different algorithmic settings, it is important to note that community labels are arbitrary. Most methods return a partition of the nodes, but the numerical identifiers assigned to each community (0, 1, 2, \dots) can be permuted without changing the partition itself. In our visualizations, however, each community is mapped to a color. Therefore, two very similar partitions can appear very different simply because the same community has been assigned a different label, which makes interpretation and comparison across figures a bit confusing.

To aleviate this problem, we have aligned community labels before plotting. We use the functions \texttt{match\_ communities} and \texttt{reorder\_communities} (see \texttt{utils.py}) to match each community in a target partition to the most similar community in a reference partition. Similarity is quantified through the Jaccard overlap between node sets, $J(A,B)=\lvert A\cap B\rvert/\lvert A\cup B\rvert$, and communities are paired greedily while enforcing a one-to-one assignment. The target partition is then reordered according to this mapping so that, whenever possible, the same community is displayed with the same color across networks. This heuristic does not guarantee a perfect alignment, but it results in more consistent plots.

Since this alignment step is only meant to improve the readability of the visualizations (and was not one of the main objectives of the practice), parts of the implementation supporting this functionality were generated with AI assistance and then reviewed and adapted to our use case.

In the primary school experiment, we apply the same idea but using the ground-truth school groups as reference. In Figure~\ref{fig:primary_school_communities}, the detected communities (unweighted and weighted) are therefore relabeled to best match the original classes, so that colors are more consistent. A consequence of this alignment is that community identifiers become non consecutive (some labels in the reference have no matched detected community), which explains why the labels shown in Figure~\ref{fig:primary_school_communities} are not consecutive. For the same reason, Figures~\ref{fig:community_composition_unweighted} and~\ref{fig:community_composition_weighted} contain white ``empty bins'', as they correspond to label positions that remain unused after relabeling.

\section{Conclusion}

In this report, we have analyzed the performance of different community detection algorithms on both synthetic and real-world networks, highlighting the strengths and limitations of each approach.

In the first task, using stochastic block model networks, we have observed that all three algorithms (Infomap, Louvain, and Agglomerative Greedy) successfully recover the ground-truth community structure when the intra-block connectivity is sufficiently high. However, their behavior differs significantly in sparse regimes. Methods based on modularity tend to detect artificial communities even in random-like networks, producing relatively high modularity values despite a meaningful structure missing. In contrast, Infomap is more conservative in these cases, often detecting a single community when no clear flow-based structure is present. As the network density increases, standard evaluation metrics (Jaccard, NMI, and NVI) confirm that Infomap is the most sensitive and fastest to perfectly capture the true underlying blocks. It is closely followed by Louvain, while the Greedy algorithm requires much denser internal connections to completely eliminate misclassifications. These results emphasize that modularity alone is not a reliable indicator of true community structure and should always be complemented with additional validation metrics.

In the second task, the analysis of a real-world school contact network has shown how community detection algorithms can also be used to determine the underlying social organization of a group of people. The Louvain algorithm has identified communities at the grade level rather than the class level, revealing that interaction patterns are stronger within grades than within individual classes. Furthermore, the comparison between weighted and unweighted networks has been useful to understand how edge weights play can be important when improving the resolution of detected communities.
